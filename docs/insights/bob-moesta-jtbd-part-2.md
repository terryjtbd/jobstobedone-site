# Bob Moesta 谈待办任务（下）：把客户转换变成产品与增长

> 经典解读｜根据 Bob Moesta 与 Des Traynor 的访谈整理<br>
> 原访谈：Des Traynor，Intercom<br>
> 核心受访者：Bob Moesta<br>
> 说明：本文为访谈观点的中文整理与解读，不是 Terry 原创理论文章。文中 Jobs to Be Done 统一译为“待办任务”。

> 阅读提示：上集讨论客户为什么会离开旧办法、转而雇佣新方案；本文继续讨论，企业如何把这种转换洞察变成产品、营销与增长决策。

在上集中，我们把待办任务放回客户的转换时间线：现状为什么难以继续忍受，新方案承诺了什么进步，旧习惯和新方案焦虑又怎样阻止改变。

但理解客户为什么转换，只是开始。企业真正面对的难题是：访谈里有大量故事、细节和情绪，怎样才能不把它们简化为零散“痛点”，而是形成能指导产品与增长的判断？

Bob Moesta 的回答可以概括为：先界定客户所在的场景与想取得的结果，再用真实转换案例找出其中反复出现的因果机制。

> 场景 × 期望结果 = 待办任务

这不是一个用于套公式的模板，而是一种把抽象需求重新落回客户现实处境的方法。

## 一、待办任务始于场景，而不是用户标签

企业经常以人群给市场命名：年轻人、宝妈、中小企业、管理者、高净值客户。这些分类有时有助于投放或渠道选择，但它们并不天然解释客户为什么会在某一刻采取行动。

同一个人，在不同情境中会雇佣完全不同的方案。一位管理者可能在日常工作中继续沿用旧的协作工具；但当一个跨城市项目进入关键交付期、任务反复丢失、协调成本急剧上升时，他或她开始寻找新方案。

此时真正影响决策的，不是“管理者”这个身份，而是“项目进入关键期且现有协作方式失效”这一场景。

场景不是一句笼统的使用描述，而是包含了足够具体的约束：

- 正在发生什么变化或触发事件？
- 客户原来依靠什么办法处理？
- 哪项进展变得重要，且不能再被拖延？
- 时间、预算、技能、关系或风险有哪些现实限制？

当场景清楚后，期望结果才不会被写成“更方便”“更高效”这类无边界的形容词。

## 二、期望结果不是功能指标，而是客户想取得的进步

产品团队常把“结果”理解成可直接测量的产品指标，例如更快完成、少点几次、提高留存。这些指标可能重要，但它们仍然是企业或产品视角的度量方式。

待办任务所说的期望结果，首先是客户在其情境中希望获得的进步。

例如，一位刚搬进新居的客户去 IKEA，不只是为了“买到一张桌子”或“以更低价格购买家具”。在有限预算、有限时间、缺少搭配经验的场景中，他或她可能希望取得的进步是：尽快、可负担地把一个尚未安顿好的住处变成能够生活、能够招待他人的家。

这里包含了多层结果：

- **功能上**，空间能够被使用，家具能带回去并装起来；
- **情感上**，减少无从下手和不断拖延的焦虑；
- **社会上**，新家不再显得临时、凌乱或缺乏体面。

如果企业只把结果理解为“客户购买了家具”，就会错过客户同时在雇佣的路径设计、陈列灵感、自助配送、可组装性、餐饮休息以及价格可预期性。客户雇佣的是一套帮助自己完成安家进步的方案。

因此，待办任务不是产品功能的别名，也不等于客户口中的一个愿望。它要求企业把“客户要什么”继续问下去：在什么处境下，为了取得什么进步？

## 三、不要停留在表面理由，要寻找转换背后的因果机制

访谈中，客户常常给出看似明确的理由：“因为价格更低。”“因为朋友推荐。”“因为界面更好看。”“因为功能更全面。”这些可能都是真的，但如果访谈到此结束，企业得到的只是事后解释，而不是能够复用的洞察。

例如，“朋友推荐”通常不是完整原因。需要继续追问：为什么当时会相信这位朋友？客户原来正在犹豫什么？推荐提供了什么证据，降低了哪一种焦虑？如果没有这次推荐，客户会继续拖延、选择其他方案，还是回到旧办法？

Bob Moesta 的访谈方法，强调从事件链中寻找因果：

| 访谈层次 | 看见的内容 | 需要继续追问的方向 |
| --- | --- | --- |
| 表面陈述 | “我觉得它更方便” | 是什么不方便的具体情境促使你寻找替代方案？ |
| 触发事件 | “项目出了问题” | 这个问题带来了什么后果，为什么当时不能继续忍受？ |
| 比较与权衡 | “我也看过其他产品” | 哪些选项被排除？它们在哪一点无法支持你取得进步？ |
| 焦虑与风险 | “我担心迁移麻烦” | 什么证据、试用或支持最终让你相信风险可控？ |
| 取得进步 | “后来工作顺了很多” | 哪个结果改变了你的行为、情绪或他人对你的评价？ |

研究者的工作不是替客户补充一个漂亮的解释，而是让时间顺序和具体事件自己显露机制。这也是为什么“为什么”不能被连续追问成抽象哲学问题。每一次追问都应回到一个具体时刻、一个选择、一个后果：当时发生了什么？你接着做了什么？为什么这件事改变了你的判断？

## 四、从故事到模式：用编码而不是直觉做判断

单个客户故事很有感染力，却不能直接代表市场。待办任务研究也不是搜集几个生动案例，然后挑一个最符合产品想象的故事。

Bob Moesta 提到访谈编码的重要性：把多场转换访谈中反复出现的事件、情境、语言、顾虑与结果进行整理，寻找模式，而不是依赖研究者的记忆与直觉。

可以从四类内容开始编码：

| 编码对象 | 需要记录什么 | 可能帮助识别的模式 |
| --- | --- | --- |
| 情境与触发 | 何时、何地、发生了什么、旧办法为何失效 | 哪些时刻最容易产生转换压力 |
| 期望进步 | 客户试图改变的实际结果、情感状态、社会处境 | 客户真正雇佣方案要完成什么 |
| 拉力与阻力 | 现状推动、新方案吸引、惯性、焦虑 | 转换为何发生或为何停滞 |
| 证据与结果 | 客户相信什么、试用了什么、最终如何验证 | 如何降低焦虑并缩短决策路径 |

编码的目的不是制造一个看起来很科学的表格，而是检验：不同客户的转换中，是否存在稳定、可解释、可行动的共同机制。

例如，十位客户都提到“价格”，并不意味着低价就是待办任务。可能其中八位真正关心的是预算不可失控，两位关心的是避免决策失误，还有一些人说价格只是为了合理化早已形成的偏好。只有放回场景、结果与转换过程，价格的含义才会变得清楚。

## 五、结果导向创新：产品不是被设计出来的，而是被雇佣来取得进步

当企业把待办任务定义为“场景中的期望进步”，产品开发的重心会发生变化。

传统问题通常是：我们还能增加什么功能？结果导向的提问则是：在这个场景中，客户正在努力取得什么进步？现有方案在哪些环节让这个进步变慢、变难、变不可靠或风险过高？

两者的差别很大。前者容易得到功能清单；后者会迫使团队理解完成整个进步路径中的障碍。

产品机会未必只在核心功能中，也可能出现在客户开始行动之前、使用过程中或确认结果之后。

仍以“团队更换协作工具”为例：如果客户的待办任务是“在跨部门、高压交付的阶段，让团队重新获得可靠的推进节奏”，机会可能包括：

- 让历史资料和任务快速迁移，降低切换焦虑；
- 让不同角色立刻理解新的协作方式，避免学习成本阻碍启动；
- 让负责人能及时看到风险与阻塞，而不是只获得更多报表；
- 让团队成员知道接下来该做什么，减少协调与重复确认。

这些不一定都表现为一个新按钮。它们可能涉及上手流程、模板、服务支持、信息架构、定价、交付与营销承诺。

因此，待办任务并不要求企业先把产品定义得更复杂；恰恰相反，它要求企业把注意力集中在客户取得进步所需的关键步骤上，删去对该进步无关紧要的复杂性。

## 六、营销不是产品完成后的包装，而是帮助客户相信“可以改变”的一部分

许多团队把产品和营销分开：产品负责创造价值，营销负责把价值讲出去。

但从客户转换的角度看，营销并不是产品完成之后的外部包装。客户在转换前存在焦虑，营销、销售、试用、案例、演示、承诺和服务，都会直接影响客户是否相信自己能安全地取得进步。

因此，营销的任务不是泛化地说“我们更好”，而是帮助特定场景下的客户回答四个问题：

1. 你是否理解我眼下遇到的具体处境？
2. 你能否帮助我取得我真正想要的结果？
3. 你和我已经在用的旧办法相比，差别在哪里？
4. 我怎样能以足够低的风险开始尝试？

如果营销只强调功能，客户仍可能觉得“听起来不错，但不适合我现在的处境”；如果营销只讲愿景，却不能提供可信证据，也无法降低转换焦虑。

产品与营销应共同完成一件事：让客户看见进步的可能性，并有把握跨出转换的第一步。

## 七、Push 与 Pull：增长不是单纯加大推动力

在访谈中，Bob Moesta 区分了 Push 与 Pull。

**Push** 是企业推动客户行动的力量，例如广告触达、销售跟进、促销、提醒、渠道覆盖。

**Pull** 是客户从自身处境出发，主动被“取得进步的可能性”吸引而来的力量。

企业当然需要 Push，但如果没有 Pull，增长只能持续依赖更大的投入。客户可能因为优惠、压力或暂时好奇而尝试，却不一定真正完成转换，也不一定持续雇佣。

真正有力量的 Pull，通常来自三个要素同时成立：

- 客户已经感受到现状的压力，不能再无限期拖延；
- 新方案清楚承诺了客户在意的进步，而不是泛化优势；
- 企业给出了足够可信的证据和低风险的第一步，降低客户焦虑。

这也是待办任务对增长的意义：它不把增长只看成“让更多客户进入漏斗”，而是理解哪些客户正处于可能转换的场景，以及企业如何帮助他们完成这次转换。

## 结语：待办任务连接洞察、产品与增长

待办任务不是一个只属于用户研究部门的概念。它从客户转换开始，但会一路影响：企业定义的是谁的什么问题、产品优先解决什么、营销应该说什么、试用如何设计、销售怎样降低焦虑、服务怎样帮助客户确认进步。

这条链路可以概括为：

> 识别客户发生转换的场景 → 理解客户试图取得的进步 → 定义待办任务 → 找出完成进步的阻碍 → 设计产品、信息与服务 → 帮助客户跨过焦虑并确认结果

因此，“场景 × 期望结果 = 待办任务”并不是一句用来贴在墙上的定义。它要求企业持续回到客户的真实生活与工作中，观察他们何时开始无法忍受旧办法，又如何为取得进步而重新组织选择。

只有这样，待办任务才不会被误用成另一种需求标签，而会真正成为连接用户洞察、产品创新与商业增长的起点。
